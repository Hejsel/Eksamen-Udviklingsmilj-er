Hvad er Git?

1. Git er et distribueret versionsstyringssystem (VCS), der giver udviklere mulighed for at spore ændringer, samarbejde effektivt og arbejde offline. I modsætning til traditionelle systemer gemmer Git snapshots af projektets tilstand.

2. Git er et kraftfuldt versionsstyringssystem (VCS), der bruges til at spore ændringer i filer og facilitere samarbejde i softwareprojekter. Git adskiller sig fra traditionelle versioneringssystemer ved at tage snapshots af dit projekt i stedet for at gemme forskelle mellem filer. Det betyder, at hver commit repræsenterer en komplet tilstand af projektet på det tidspunkt.

Nøglefunktioner:

  • Snapshots: Hele projektet gemmes som billeder af dets tilstand.
  • Distribueret: Hver udvikler har en komplet kopi af projektet.
  • Sikkerhed: Checksums forhindrer utilsigtede ændringer.
  • Offline arbejde: Lokale handlinger kræver ikke en netværksforbindelse.

Snapshots vs. Differences

1.Git adskiller sig fra traditionelle versionsstyringssystemer ved at gemme snapshots af hele projektet i stedet for forskelle mellem filer.

2. Snapshots, ikke forskelle: Traditionelle systemer som Subversion gemmer ændringer mellem filer (delta-baseret versionering). Git gemmer derimod snapshots af hele projektets tilstand. Hvis en fil ikke ændres, opretter Git blot en reference til den tidligere version.

Centraliseret vs. Distribueret Versionsstyring

1. Git fungerer som et distribueret system, hvor hver udvikler har en komplet historik. Dette står i kontrast til centraliserede systemer, der afhænger af en enkelt server.

2. Næsten alle handlinger er lokale: Git kræver normalt ikke en netværksforbindelse for at udføre de fleste operationer. Dette gør det muligt at arbejde offline, hvilket giver stor fleksibilitet og hastighed.

De Tre Tilstande i Git

Filer i Git kan være i tre hovedtilstande: Modified, Staged og Committed. Disse tilstande repræsenterer, hvordan ændringer flyttes gennem Git-systemet.

  • Modified: Filer er ændret, men endnu ikke klar til commit.
  • Staged: Ændringer er markeret til commit.
  • Committed: Ændringer er permanent gemt i Git-databasen.

Fordele ved Git:

  • Høj hastighed: Lokale operationer er næsten øjeblikkelige, da historikken gemmes på din maskine.
  • Integritet: Alt i Git er checksummet, hvilket betyder, at intet kan ændres uden, at Git opdager det.
  • Sikkerhed: Det er svært at miste data, da næsten alle operationer kun tilføjer data til databasen.

Git Cheat Sheet

En komplet guide til versionsstyring og samarbejde med Git

1. Git Konfiguration

Git-konfiguration er det første skridt, når du begynder at arbejde med Git. Det hjælper med at identificere, hvem der laver ændringerne, og sikrer, at dine commits bliver korrekt mærket. Konfigurationen inkluderer indstillinger som dit navn, din e-mail og aktivering af terminalfarver. Disse indstillinger er globale, hvilket betyder, at de gælder for alle dine Git-projekter på maskinen.

Hvorfor er det vigtigt? Når du arbejder på teams eller open source-projekter, gør korrekt konfiguration det nemt for andre at se, hvem der har lavet hvilke ændringer. Desuden forbedrer aktiveringen af farver i terminalen læsbarheden af kommandooutput.

Opsætning af dit navn 
# Opsæt dit navn
git config --global user.name "Dit Navn"

Forklaring: Dit navn vil blive knyttet til hver commit, du laver, hvilket gør det nemt for andre at identificere dine ændringer.
Billedsprog: Forestil dig, at dit navn står som afsender i en fælles notesbog.

Opsætning af din e-mail 
# Opsæt din e-mail
git config --global user.email "dinemail@example.com"

Forklaring: Din e-mail bruges til at identificere dig i Git-historikken og er essentiel ved samarbejde.
Billedsprog: Det er som at tilføje dit visitkort i en rapport, så andre kan kontakte dig.

Aktivering af farver i terminalen 
# Aktivér farver i terminal-output
git config --global color.ui auto

Forklaring: Denne indstilling aktiverer farvekodning, hvilket gør Git-output mere læsbart i terminalen.
Billedsprog: Det svarer til at bruge farvede overstregningstuscher for at fremhæve nøglepunkter i dine noter.

2. Start Et Projekt

Når du starter et projekt med Git, kan du enten initialisere et helt nyt repository eller klone et eksisterende. Initialisering skaber en lokal "dagbog" for at holde styr på ændringer, mens kloning kopierer en eksisterende "dagbog" med hele dens historik.

Hvorfor er det vigtigt? Initialisering er nødvendigt for at starte versionsstyring af filer. Kloning er afgørende, når du vil samarbejde om eksisterende projekter eller bruge fælles skabeloner.

Initialisering af et nyt repository 
# Initialiser et nyt repository
git init

Forklaring: Opretter en ny Git-dagbog (repository) i den aktuelle mappe. Dette gør det muligt at begynde at spore ændringer i projektet.
Billedsprog: Det er som at starte en helt ny notesbog for dine idéer.

Kloning af et eksisterende repository 
# Klon et repository fra en URL
git clone [repository-URL]

Forklaring: Kopierer en eksisterende Git-dagbog inklusive hele dens historik til din lokale maskine.
Billedsprog: Det svarer til at få en kopi af en vens notesbog med alle deres noter.

3. Staging og Snapshot

Staging-processen i Git hjælper dig med at vælge specifikke ændringer, der skal inkluderes i den næste "snapshot" eller commit. Dette giver dig kontrol over, hvilke filer eller ændringer der bliver gemt.

Hvorfor er det vigtigt? Med staging kan du undgå at inkludere ufuldstændige eller utilsigtede ændringer i en commit. Dette gør det nemmere at holde din Git-historik ren og organiseret.

Tilføj ændringer til staging 
# Tilføj specifikke filer til staging
git add [filnavn]

Forklaring: Flytter ændringer i specifikke filer til staging-området. Disse ændringer vil blive inkluderet i den næste commit.
Billedsprog: Det svarer til at lægge bestemte sider fra din notesbog i en mappe, der skal gemmes.

Se status på ændringer 
# Se status på dit repository
git status

Forklaring: Viser, hvilke filer der er ændret, tilføjet til staging eller klar til commit.
Billedsprog: Det er som at tjekke en pakkeliste, før du sender noget af sted.

4. Commit

En commit gemmer ændringer i projektets historik. Hver commit markerer en ny milepæl og hjælper med at holde styr på projektets udvikling.

Hvorfor er det vigtigt? Ved at bruge commits kan du vende tilbage til tidligere versioner og dokumentere ændringer. Dette gør det nemmere at samarbejde og fejlfinde.

Opret en commit 
# Lav en commit med en besked
git commit -m "Din besked"

Forklaring: Gemmer de ændringer, der er i staging-området, som en ny version i historikken. Beskeden beskriver ændringerne.
Billedsprog: Tænk på det som at tage et billede af dit projekt og skrive en note om, hvad der blev ændret.

Se commit-historik 
# Vis historik over commits
git log

Forklaring: Viser en liste over alle commits, deres hash-id'er, beskeder og tidsstempler. Dette hjælper med at forstå projektets udvikling.
Billedsprog: Det er som at bladre igennem en logbog med detaljerede noter om hver ændring.

5. Branching og Merging

Branching giver dig mulighed for at arbejde på forskellige funktioner eller dele af projektet uden at påvirke hovedlinjen. Når arbejdet er færdigt, kan du merge det tilbage.

Hvorfor er det vigtigt? Ved at oprette grene kan flere personer arbejde parallelt uden konflikt. Merging samler det hele igen, når arbejdet er klart.

Opret en ny branch 
# Opret en ny branch
git branch [branch-navn]

Forklaring: Skaber en ny gren baseret på den nuværende version af projektet.
Billedsprog: Det er som at starte en ny linje i en notesbog for en specifik idé.

Skift til en branch 
# Skift til en eksisterende branch
git checkout [branch-navn]

Forklaring: Flytter dit arbejdsområde til den valgte branch.
Billedsprog: Det svarer til at åbne en specifik side i notesbogen for at arbejde på den idé.

Merge en branch 
# Flet en branch ind i den nuværende
git merge [branch-navn]

Forklaring: Kombinerer ændringer fra en anden branch ind i den nuværende branch.
Billedsprog: Det er som at samle noter fra to sider i din notesbog til én.

Slet en branch 
# Slet en branch
git branch -d [branch-navn]

Forklaring: Fjerner en gren, der ikke længere er nødvendig, fra projektet.
Billedsprog: Det svarer til at rive en side ud af notesbogen, fordi den idé er afsluttet.

6. Remote Repositories

Remote repositories giver dig mulighed for at synkronisere dit arbejde med fjernservere, hvilket gør det muligt at samarbejde med andre og sikkerhedskopiere dit arbejde.

Hvorfor er det vigtigt? Fjernservere er essentielle for teamsamarbejde og sikrer, at alle arbejder på den samme version af projektet.

Tilføj en fjernserver 
# Tilføj en fjernserver
git remote add origin [url]

Forklaring: Forbinder dit lokale repository med en fjernserver, så du kan dele dit arbejde.
Billedsprog: Det svarer til at tilknytte din notesbog til en sky for sikkerhedskopiering.

Push ændringer 
# Upload ændringer til fjernserveren
git push origin main

Forklaring: Sender dine commits fra det lokale repository til fjernserveren.
Billedsprog: Det er som at uploade filer til en delt mappe.

Pull ændringer 
# Hent ændringer fra fjernserveren
git pull origin main

Forklaring: Opdaterer dit lokale repository med ændringer fra fjernserveren.
Billedsprog: Det er som at downloade de nyeste noter fra en delt mappe.

7. Konflikthåndtering

Konflikter opstår, når flere personer ændrer de samme linjer i en fil. Git hjælper dig med at identificere og løse disse konflikter.

Hvorfor er det vigtigt? Effektiv konflikthåndtering sikrer, at ingen ændringer går tabt, når flere arbejder på det samme projekt.

Løs konflikter 
# Rediger konfliktmarkeringer i filen
# Marker med <<<<< og =======
git add [filnavn]
git commit -m "Konflikt løst"

Forklaring: Konfliktmarkeringer angiver, hvor ændringer overlapper. Rediger filen for at beholde de ønskede ændringer og commit.
Billedsprog: Det svarer til at samle noter fra to forskellige personer og flette dem sammen til én tekst.

8. Ignorering af Filer

Brug en `.gitignore`-fil til at ekskludere specifikke filer eller mapper fra Git. Dette er nyttigt til midlertidige filer eller følsomme data.

Hvorfor er det vigtigt? Ved at ignorere irrelevante filer kan du holde dit repository organiseret og undgå at inkludere uønskede data.

Eksempel på .gitignore 
# Ignorer logs og midlertidige filer
logs/
*.tmp
.env

Forklaring: Denne .gitignore-fil udelukker mapper som `logs/`, filer med `.tmp`-endelse og `.env`-filer fra versionering.
Billedsprog: Det er som at sætte et "Ikke til opbevaring"-mærke på ting, du ikke ønsker at gemme i din notesbog.

4. Git Kommandoer

I Git har du mange værktøjer til at administrere ændringer og snapshots. Disse kommandoer hjælper dig med at styre, uploade og sammenligne ændringer effektivt.

Hvorfor er de vigtige? Kommandoer som git reset, git diff, og git commit giver dig kontrol over, hvordan ændringer administreres i dit projekt.

Fortryd staging af filer 
# Fjern filer fra staging
git reset [file]

Forklaring: Fjerner en fil fra staging-området uden at miste ændringer i working directory.
Billedsprog: Det svarer til at tage noget ud af en pakkeliste, men stadig have det klar til næste forsendelse.

Se ændringer, der ikke er staged 
# Vis ændringer siden sidste commit
git diff

Forklaring: Viser forskelle mellem working directory og den seneste commit. Dette hjælper med at se, hvad der er ændret, men ikke staged.
Billedsprog: Det er som at sammenligne to udgaver af et udkast for at se, hvad der er tilføjet eller slettet.

Se ændringer, der er staged 
# Vis ændringer i staging-området
git diff --staged

Forklaring: Viser forskelle mellem staging-området og den seneste commit. Dette hjælper med at verificere, hvad der bliver committed.
Billedsprog: Det svarer til at dobbelttjekke indholdet af en pakkeliste, før du sender pakken af sted.

Lav en commit med en besked 
# Commit staged ændringer med en besked
git commit -m "[beskrivende besked]"

Forklaring: Gemmer de ændringer, der er staged, som et snapshot i Git-historikken, og tilføjer en besked for at beskrive ændringerne.
Billedsprog: Det er som at gemme en ny version af en rapport og skrive en note om, hvad der blev ændret.

6. Remote Repositories

Synkroniser dit arbejde med fjernservere:

git remote add origin [url]
git push origin main
git pull origin main

Tip: Fjernservere gør det muligt at dele og samarbejde på projekter.

7. Konflikthåndtering

Løs konflikter, når flere ændrer samme filer:

# Rediger konfliktfiler markeret med <<<<< og =======
git add [filnavn]
git commit -m "Konflikt løst"

Forklaring: Læs konfliktmarkeringerne og vælg de ønskede ændringer.

8. Ignorering af Filer

Udelad specifikke filer fra versionsstyring:

# Eksempel på .gitignore
logs/
*.tmp
.env

Tip: Brug `.gitignore` for midlertidige filer eller følsomme data.
